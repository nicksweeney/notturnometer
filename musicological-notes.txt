Musicological notes — close calls in work identity
===================================================

A log of the judgment calls behind work-grouping decisions that the
canonicalization rules can't fully automate: the cases where two recordings or
titles are arguably the same work and arguably distinct, and a human had to
decide. Each entry records the case, the evidence (favour the performance-PID
fingerprint over title strings), the decision, the reasoning, and any rule it
sets or tests.

PROCESS: musicological close calls get an adversarial review from "Cerys" (a
harp-scholar persona; spawn the `cerys` agent) BEFORE the decision is finalised.
Pose the PURELY musicological hypotheses, stripped of database framing; demand
pushback. Record their verdict in the entry. (Cerys overturned the instrument-
family rule below and supplied the transcription rubric.)


-------------------------------------------------------------------------------
Transcription depth as the work boundary  (violin/cello vs keyboard variants)
2026-06-15   [supersedes the instrument-family rule, overturned same day]
-------------------------------------------------------------------------------

Case. Franck, Sonata in A major (M.8 / FWV.8). Composed for violin (1886);
Delsart's cello transcription (1888, Franck-sanctioned) is the standard cello
version. In the corpus it fragments 11 ways / 226 airings, on three axes:
catalogue (M.8 / FWV.8 / none), spelling, and instrument (violin / cello /
"violin or cello" / "either violin or cello").

Evidence (performance PIDs). 18 distinct recordings: 11 violin, 7 cello. The
performer credit is ALWAYS unambiguous — a cellist plays cello even under a
"violin or cello" title — whereas the segment title is not reliable for the
instrument. Durations overlap completely (violin 1530-1815s, cello 1672-1787s):
one ~27-minute work. So the instrument can only be recovered from the performer
fingerprint, never from the title.

First call (Nick, OVERTURNED). Instrument FAMILY is the boundary: keyboard
(organ/piano/harpsichord/fortepiano) interchangeable for one player -> MERGE;
non-keyboard (violin/cello/winds) -> SPLIT, since a cellist doesn't play violin.
M.8 would split violin/cello, deferred to the fingerprint oracle.

OVERTURNED (Cerys adversarial consult, 2026-06-15). The instrument-family line
"isn't tracking anything musical": harpsichord->piano changes voicing/dynamics
MORE than violin->cello; it would fragment the catalogue (the Arpeggione sonata,
Brahms's own viola versions of the clarinet sonatas, the Bach cello suites all
"different works"); and the harp breaks it (harp transcriptions of Debussy piano
works are neither keyboard nor orchestral string/wind). The real axis is DEGREE
OF RECOMPOSITION, independent of instrument:

  POLICY (adopted). Work identity = transcription depth, three tiers:
    - LITERAL transcription (same notes, new instrument) -> SAME work, version tag.
      (Delsart cello Franck; Brahms's own viola versions; Bach suites across the
      string family; Haydn keyboard-vs-piano trios.)
    - Composer's OWN re-scoring for new forces -> same work, FLAGGED. (Brahms
      2-piano vs orchestral Haydn Variations; Bach BWV1058/1042; Ravel's Pictures
      is a third-party orchestration = still same work, orchestrator is a credit.)
    - FREE PARAPHRASE / fantasy -> a NEW, distinct work. (Liszt operatic
      paraphrases, Busoni-Bach.) Compositional intervention is the boundary.

Consequences. The keyboard merges already shipped (54937e7, Haydn/Scarlatti) STAND
— "same notes, different keyboard" is a literal transcription = same work under
the new rule too; no reversal. M.8 (Franck Sonata in A) cello version MERGES with
the violin original (straight Delsart transcription) — the violin/cello split is
DROPPED, the oracle is no longer needed for it. M.8 untangle SHIPPED 2026-06-15
(ed48b9a): 10 aliases fold all 11 keys onto "Violin Sonata in A major, M.8",
one 226x work; the cello version is a display facet.

OPERATIONAL RUBRIC (Cerys, for a system seeing only title + performer + duration).
Key insight — the "advertising asymmetry": deep recompositions ADVERTISE
themselves (the marker is the whole selling point of a paraphrase), while shallow
transcriptions go unmarked. So DEFAULT TO MERGE; split-markers are the active
ingredient. "Look for a reason to split; absent a marker, merge." Decision order:
SPLIT -> ESCALATE -> else MERGE.

  SPLIT (new distinct work):
    - "paraphrase" / "concert paraphrase" / "réminiscences de" — gold.
    - "fantasy/fantasia/fantaisie/Fantasie" + "on/über/sur" + a named WORK — the
      PREPOSITION carries it; bare "Fantasia" is just a form-title -> merge.
    - "concert transcription / transcription de concert" — the "concert" qualifier
      signals showpiece recomposition.
    - "freely after / frei nach / librement" — the ADVERB "freely" is the trigger
      (bare "after"/"based on"/"d'après"/"nach" is overloaded -> ESCALATE).
    - "on themes from <named work>" + a virtuoso solo instrument in the credits.
  MERGE (same work, version is a display facet — the DEFAULT):
    - NO recomposition marker -> merge under the stated composer.
    - "arr." / "transcr." / "orch." / "version for" / "for <instruments>" /
      "realised" / "ed./edition" (and NOT "concert"/"freely").
    - Composer's own re-scoring -> merge, flag. Different duration/instruments are
      DISPLAY FACETS, never a work split.
    - *** BORROWED-THEME GUARD (drill this in): the COMPOSER FIELD is the author;
      a theme-source named in the title is NEVER the author and never the filing
      home. "variations/rhapsody on a theme OF/BY <a PERSON>" (Brahms Haydn/
      Paganini Variations, Rachmaninov Paganini Rhapsody, Liszt Totentanz) =
      ORIGINAL work -> merge under the credited composer, NEVER file as an excerpt
      of the theme-source. Tell: prepositional object is a PERSON -> borrowed theme
      (merge); object is a WORK/opera -> paraphrase (consider split).
  ESCALATE (title can't see the depth): bare "after/based on" w/o "freely";
    "transcription" on a known Liszt/Busoni item; one piece under clashing EBU
    house styles; cadenzas/completions/editions (Mozart Requiem Süssmayr/Levin,
    Mahler 10, Art of Fugue); famous-composer surname used adjectivally with no
    "of". -> flag for a human, don't guess.

Downstream note (still relevant). `_strip_arrangement_tail` folding "arr./transc.
for X" -> original (371 titles / 1103 airings) is CONSISTENT with the new policy
(arrangements are same-work). No change needed there.

Barber Adagio — three works, not one  (work / version / movement in miniature)
2026-06-15
-------------------------------------------------------------------------------

Case. Barber's Adagio music appears in three guises: (a) the slow movement of the
String Quartet, Op.11 (original); (b) the Adagio for Strings — his own string-
orchestra arrangement, the famous standalone; (c) the Agnus Dei — his own choral
setting, with the Latin Agnus Dei text added. In the corpus they fragmented across
7 keys (incl. a Dover Beach Op.3 false-positive, unrelated).

Decision (Cerys consult, confirmed). THREE distinct works, no cross-guise merge:
  - String Quartet Op.11 (complete) = one work. The standalone Adagio is its slow
    movement re-scored, but a complete-quartet broadcast is the 3-movement whole —
    whole-vs-part stays split (cf. a standalone Nimrod vs the complete Enigma).
  - Adagio for Strings = one work (same musical entity as the quartet's movement,
    in a second scoring; counted in its own right as the standalone).
  - Agnus Dei = one work. Adding a sung liturgical text is NOT a mere re-scoring —
    it changes medium and genre; "the same tune with words" is the RE-TEXTING case
    = a new work (the words are constitutive). Precedent: Handel "Lascia la spina"
    / "Lascia ch'io pianga" — same melody, two vocal homes, kept distinct. The
    text is the deciding factor. (Aligns with the existing lascia-la-spina rule.)

Action. Only consolidated rephrasings WITHIN each guise (Adagio 145→152x; Agnus
Dei 54→55x; Quartet 4→7x). The three works stay distinct.

Spurious / misattributed works — policy (NOT yet implemented)
2026-06-15
-------------------------------------------------------------------------------

Question. How should the log handle works credited to a composer who almost
certainly didn't write them (BWV 1070 → J.S. Bach; the Albinoni/Giazotto Adagio;
the "Haydn" Toy Symphony = Leopold Mozart/Angerer; the Pergolesi/Pulcinella
sources = Gallo, Wassenaer; spurious "Mozart"...)? Nick's pragmatic stance: "if
the BBC's going with it, and presumably the performers/broadcasters too, then
[shrug]?"

Verdict (Cerys consult; see cerys.txt CONSULT 7). The shrug conflates two facts:
"what was it broadcast AS?" (the broadcast) vs "who WROTE it?" (the music). The
field "going with it" is catalogue inertia + the marketing pull of the name, NOT
the field's judgement — and our own data shows it (1 of 36 BWV 1070 airings hedged
"attrib."; the unanimity is an artefact of rounding). "Preserve the source as
written" is a rule about a FIELD, not about the whole record. Following the credit
is fine for the credit field but CATASTROPHIC for the composer axis: it inflates
the canonical names (spurious Bach/Haydn/Mozart are false positives on exactly the
top-ranked composers) and HIDES the real one entirely (under "follow the BBC,"
Giazotto appears zero times while Albinoni gets the most-broadcast "baroque" piece
he didn't write a bar of). The moment you COUNT by composer, "follow the BBC"
becomes active misinformation biased toward the big names.

POLICY (recorded; the layer below is DEFERRED, not built). The credit is THREE
facts → three fields:
  1. credited_composer — exactly as broadcast. PRESERVED, never overwritten.
  2. attribution_status — attributed / doubtful / spurious / misattributed.
  3. likely_composer — scholarly best candidate where one exists (null otherwise).
Composer-axis analytics default to likely_composer else credited_composer, as a
VISIBLE TOGGLE ("as broadcast" vs "as authored"). The deciding line is NOT
"universal vs contested" (universality is the artefact) but "does a defensible
ALTERNATIVE authorship statement exist?" — three buckets: (A) settled
misattribution → real likely_composer (Giazotto, Leopold Mozart, Gallo/Wassenaer);
(B) doubtful-with-suspect → suspect + 'doubtful' (BWV 1070 → W.F. Bach); (C)
irreducibly anonymous → null. Keep contradictory credits ("attrib. Bach" vs
"Bach") DISTINCT in field 1 (it's evidence the credit is soft), unified in fields
2–3. The spurious/doubtful canon is a small finite curated list (low hundreds).

BWV 1070 facts (corpus, 2026-06-15). Credited to J.S. Bach in 35/36 airings — both
lineages, segment composer_mbid = 24f1766e-9635-4d58-a4d4-9413f9f98a4c (the J.S.
Bach entity). 1 tracks airing reads "attrib. Bach, Johann Sebastian" and does NOT
fold into the JS Bach group (keys to its own 'attrib bach johann sebastian'). Under
the policy: status = doubtful, likely_composer = W.F. Bach; leave the credit as JS
Bach; keep the "attrib." airing distinct (do NOT fold it). No action taken now —
the 3-field attribution layer + composer-axis toggle is a deferred sub-project.

Is vocal whole-vs-part actually a problem?  (bellwether probe + Cerys verdict)
2026-06-15
-------------------------------------------------------------------------------

Question. Before building a vocal whole-vs-part oracle, IS the over-grouping
("one catalogue number swallowing the whole work + all its arias into one blob")
actually happening? Cerys nominated 4 bellwethers spanning the genres; I pulled
the real 16-yr corpus clustering for each (see cerys.txt CONSULT 8).

What the corpus shows. The big blob fear is LARGELY NOT BORNE OUT on the
warhorses — arias/choruses/songs reliably split into their own clusters:
  - Mozart Figaro K.492: arias split (Deh vieni 22, Dove sono 13, Porgi amor 6,
    Voi che sapete 4). ONE 70-airing §k492 blob, but it's ~all the OVERTURE +
    a couple of whole-opera strays.
  - Handel Messiah HWV.56: numbers split (Rejoice greatly 14, He shall feed 8…);
    the WHOLE/multi-part airings SCATTER (Part 1 / Parts 2&3 / Part 2 / Part 3 /
    Excerpts / Overture) — no honest "complete Messiah" count.
  - Schubert Winterreise D.911: HEALTHY — cycle = 1 cluster (12), every song
    splits. The _CYCLE_CATALOGUE_REFS denylist is load-bearing here.
  - Verdi Requiem: whole work fragmented 3 ways by SPELLING/scoring (Messa 12 /
    Missa 3 / Messa…for soloists, chorus and orchestra 3). Dies Irae correctly
    NOT excerpted (≈20 min, structurally inseparable into the Tuba mirum).

Cerys verdict (adversarial; pushed back hard). The excerpt-splitting "works for
the WRONG mechanism" — arias escape the catalogue path only because the BBC
titles them WITHOUT the catalogue number, so they fall to token-sort. It's a
TITLING ACCIDENT + the cycle denylist, NOT structural understanding → fragile on
un-audited works. The three faults, graded:
  (a) Mozart overture lumped with whole opera = GENUINE ERROR. Cerys REJECTED my
      "overture is defensibly part of the opera for counting" — an opera overture
      in a concert slot is a VERSION in independent use (4 min, own programming
      logic), NOT a constituent; counting 70 overture airings as "Figaro/opera"
      badly inflates the "opera on TTN" figure. The English "Overture to The
      Marriage of Figaro" (4) should MERGE with it, not split. Fix FIRST. NB this
      collides with the DELIBERATE design choice (CLAUDE.md) to exclude 'overture'
      from _movement_slug so opera-overture phrasings merge to the opera key —
      Cerys says that choice is the bug.
  (b) Messiah whole/part SCATTER = GENUINE analytical loss, but NOT an aliasing
      problem — it needs MULTI-EPISODE LINKING (Part 1 + Parts 2&3 from one source
      broadcaster across a date window = one complete Messiah; ≈4-6 over 16 yrs).
      Different, bigger machinery. The scatter paradoxically PRESERVES the
      genuine-series / lone-Part-1 / curated-"Excerpts" distinction.
  (c) Verdi Messa/Missa + scoring tail = COSMETIC, one-alias fix (but 12 vs 18
      airings matters for big-work frequency, so do it).

The conceptual gap (Cerys's headline). CATALOGUE NUMBER AS WORK IDENTIFIER vs
AS FAMILY LABEL. K.492 is a FAMILY LABEL (overture + complete opera + aria
selections + act-series); the catalogue path treats it as a work identifier and
conflates the family. Warhorses hide this because their excerpts are titled by
name without the number. The blob GENUINELY bites where the BBC attaches the
number/opus TO the excerpt — Cerys's risk list to probe next:
  - Bach cantata / Passion movements carrying BWV (e.g. "Erbarme dich" BWV.244
    merging into the whole St Matthew Passion);
  - Vivaldi RV sets (RV shared by set + individual concerto);
  - Schubert Schwanengesang D.957 (NOT a real cycle — songs less clearly "part
    of"; check the denylist fires when BBC titling is inconsistent);
  - Schumann Dichterliebe Op.48 ("Songs from Dichterliebe, Op.48" dumping 2 songs
    into the cycle cluster); 20th-c cycles (Britten Winter Words Op.52,
    Shostakovich Op.79) — no standalone-aria tradition, songs likely carry opus.

DECISION (provisional, pending Nick). Don't build the oracle blind. The cheap
next step is a TARGETED PROBE of Cerys's risk classes (do excerpts there actually
carry the number and collapse?) — that tells us whether there's a real problem
worth machinery. Concrete already-known fixes regardless: the Verdi one-alias (c,
cosmetic) and the opera-overture level-confusion (a, revisits a deliberate design
choice). The Messiah multi-episode linking (b) is a separate larger sub-project.

RISK-CLASS PROBE RESULT (2026-06-15) — ORACLE RETIRED as unmotivated. Ran the
recording_pid+duration oracle over segment_events (2012+) for Cerys's risk
classes: flag a §catalogue key whose recordings span min≤360s AND max≥1200s (a
whole work + an excerpt collapsed under one key). Totals, ENTIRE corpus:
  - Bach: 13 blob keys / 57 misgrouped excerpt-side airings — but only 14 VOCAL;
    the other 43 are INSTRUMENTAL movement-excerpts (Air on the G string folded
    into Orchestral Suite 3 §bwv1068; cello/violin-suite Sarabande/Largo folded
    into the whole) = the known _movement_slug class + a few locator-recognition
    gaps ("(Sarabande)", "- Largo", trailing movement lists), NOT vocal.
  - Vivaldi: 1 blob key / 7 airings — the Gloria RV.589 (movements "Cum Sancto
    Spiritu"/"Et in terra pax" titled "Gloria in D RV.589 - <mvt>", no locator).
  - Schubert (Schwanengesang D.957), Schumann (Dichterliebe Op.48), Britten
    (Op.52), Shostakovich (Op.79): ZERO blobs. Cerys's most-feared song-cycle
    classes DON'T collapse — the _CYCLE_CATALOGUE_REFS denylist + the fact that
    `Op` routes to the token-sort path (not the catalogue path) already handle
    them. The prediction was empirically wrong for this corpus.
The genuine VOCAL whole-vs-part collapse is ≈21 airings corpus-wide (14 Bach +
7 Vivaldi) out of ~125k. VERDICT: no vocal whole-vs-part ORACLE — unmotivated.
The residue is a HANDFUL of famous named excerpts titled with the catalogue
number but no locator (Jesu bleibt meine Freude/"Jesu Joy" BWV.147; the Sinfonia
of BWV.21; the Gloria RV.589 movements) — hand-alias / small vocal-movement-
locator extension, opportunistic, NOT an engine. The Verdi Messa/Missa one-alias
and the opera-overture level-confusion (a) remain the only concrete open items;
the Messiah multi-episode linking (b) stays parked as a separate sub-project.

Opera overture filed under the opera's catalogue number — RATIFIED (not fixed)
2026-06-15
-------------------------------------------------------------------------------

Case. An opera overture keys to the opera's catalogue ref, the SAME key as the
whole opera (Figaro overture → §k492, the Le Nozze di Figaro key). This is a
DELIBERATE design choice: `_movement_slug` excludes 'overture' so all phrasings
of an opera's overture merge to the opera key (CLAUDE.md). Cerys (vocal whole-vs-
part consult, CONSULT 8) flagged it a GENUINE level-confusion error — the
overture is a standalone VERSION in concert use, not a constituent of the staged
work; counting overture airings as "the opera" inflates the opera figure.

Measured extent (the whole archive). 13 catalogue keys mix an overture title with
a non-overture title. They are TWO unrelated classes:
  - BAROQUE SUITE-"OVERTURES" (the big numbers, NOT opera): §bwv1066 (42 ov + 81
    suite), §bwv1068, Telemann La Bizarre, Handel HWV.351. Here "Ouverture" is
    Bach's GENRE name for the orchestral suite (or its opening French-overture
    movement) — the overture IS the suite / its proper first movement; merging is
    CORRECT. Splitting these would be a bug.
  - GENUINE OPERA overtures: §k492 Figaro (69 ov + 1 whole), §k50 Bastien (9+1),
    §k384 Entführung (9+1), Vivaldi §rv719/725 (2-6). Each is ~entirely the
    overture with at most ONE whole-opera airing absorbed; §k490 is a data quirk
    (2 Figaro overtures mis-typed "K490" colliding with the K.490 concert aria).

The reframe. The feared INFLATION DOES NOT MANIFEST: whole operas essentially
never air on TTN (it programmes concert repertoire), and the cluster DISPLAYS as
the overture (most-common title wins → "Overture to Le Nozze di Figaro", not "Le
Nozze di Figaro"). So §k492 already IS the overture unit Cerys wanted; the real
harm is ~3-5 absorbed whole-opera airings across 16 years.

DECISION: RATIFY current behaviour (Nick, 2026-06-15). No code change. The
principled split is high-blast-radius catalogue-path surgery AND requires telling
an opera overture (Type A) from a Baroque suite-overture (Type B/C) from a title
string — ~85% accurate, and the 15% error on the shared keying path isn't worth
3-5 airings. Cerys CONCEDED on the measurement (CONSULT 9): "ratify; the display
heuristic already gives the practically correct answer."

Caveats Cerys asked to record (not fix):
  - The one question ratified behaviour answers WRONGLY: "what operas aired, and
    how many times?" → "Figaro 70" is the overture, not the opera. But TTN barely
    plays operas, so it's a wrong answer about a near-empty set.
  - FRAGILITY: "displays as the overture" is stable ONLY while overtures dominate
    the cluster by count. If the corpus ever covered a complete-opera-broadcasting
    era, the same key would SILENTLY start displaying as the full opera title with
    no code change. Data-dependent, not visible in the implementation.
  - TYPE A / B / C distinction (opera-overture curtain-raiser / ouverture-as-genre
    / suite-opening-movement) is real and matters for any FUTURE genre axis or
    "opera content" filter — encode it then via a composer×decade lookup, NOT a
    title-string rule. Future work, not a current defect.
